\section{Description of Implemented Algorithms}
\label{sec:algoritms}

In this section, we detail the implementation of the three algorithms used in the comparative study, highlighting how they manage 
particular cases (sparse graphs) and memory constraints.

\subsection{Brute Force (Exhaustive Search)}
The implementation uses the \texttt{std::next\_permutation} function from the C++ standard library to systematically generate all possible routes.

\begin{itemize}
    \item \textbf{Workflow:} The algorithm fixes node $0$ as the starting point and generates all permutations of the remaining $n-1$ nodes.
    \item \textbf{Error Handling:} To prevent \textit{overflow} errors, we used a \texttt{SAFE\_INT} constant. If an edge is missing or 
    the path is invalid, the cost is set to this value, and the sum is not incremented beyond the limit.
    \item \textbf{Complexity:} $O(n!)$, being limited by the total number of permutations.
\end{itemize}

\subsection{Held-Karp (Dynamic Programming and Bitmasking)}
The Held-Karp algorithm represents a significant optimization, reducing the search space by memoizing partial results.

\begin{itemize}
    \item \textbf{Data Structures:} We used a DP table of type \texttt{std::vector<std::vector<int>>} with size $n \times 2^n$. 
    \item \textbf{Bitmasking:} The state of subproblems is represented by an integer (\texttt{mask}), where the $i$-th bit is 
    $1$ if city $i$ has been visited and $0$ otherwise.
    \item \textbf{Recursion:} The \texttt{totalCost} function recursively calculates the minimum path, checking at each step if the 
    remaining path is valid (\texttt{remaining < SAFE\_INT}) before updating the local optimum.
\end{itemize}

\subsection{Greedy (Nearest Neighbor)}
It is the fastest algorithm implemented, featuring a constructive approach based on local choices.

\begin{itemize}
    \item \textbf{Logic:} At each step, the algorithm searches the adjacency matrix for the nearest unvisited neighbor of the current city.
    \item \textbf{Structural Failure (Dead End):} The implemented code explicitly handles the case where there are no edges to unvisited cities. In this situation, the algorithm returns \texttt{SAFE\_INT}, signaling the impossibility of completing the tour (a fact confirmed by the results in Table \ref{tab:accuracy}).
    \item \textbf{Complexity:} $O(n^2)$, due to the two nested loops required to visit $n$ cities and search for the minimum neighbor.
\end{itemize}
